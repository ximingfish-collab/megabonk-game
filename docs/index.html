这些文件允许重构（必然会动），但导出名称、签名、类型形状必须保留。通过 .claude/skills/check-contract skill 手动校验。

1. @minigame/core 公开 API（game/core/source/index.ts）

下列导出必须始终存在：

// 类

// 类型（client 渲染依赖）

其他导出（WEAPON_STATS / fireWeapon 等）属于内部协作 API，重构期间可以删改。

2. GameInstance 类签名

export class GameInstance {
  constructor(config: GameConfig)                  // 单参数
  start(): void
  tick(): boolean                                  // true = 本局结束
  applyAction(input: InputState): void             // 单参数（无 playerId）
  getState(): GameState
  getResult(): GameResult
}

本项目是 single-player（KUBEE threejs-3d-singleplayer 模板）。重构期间 GameInstance 应当退化为薄 facade，方法体内部委托给 world + systems，但外部接口不得改变。

四、新增内容的标准流程

方案 A 重构 (Phase 1-7) 落地后，加新东西不再需要改 GameInstance.ts。数据加在 data/，行为加在 behaviors/ / ai/，工厂自动接管。

加一把武器

  1. data/weapons.ts 新增 WeaponDef (type / behaviorId / tags / level stats ref)
  2. 如无现成行为（sweepArc / forwardArrow / orbitingAxe / spreadShot / bouncingShot / lightningChain / flameAura），在 behaviors/ 加新 .ts + 注册到 behaviors/index.ts
  3. 完成。systems/weapons.ts 自动通过 behaviorId dispatch

不再需要：改 GameInstance.ts、加 fireXxx 方法、改 switch。

加一个敌人

  1. data/enemies.ts 新增 EnemyDef
    (hp / damage / speed / behavior: 'chase'|'ranged'|'charge'|'dive' / modifier? / tags?)
  2. 4 brains 已覆盖大部分用例。如需叠加 (如 necromancer 召唤)，在 ai/modifiers/ 加 EnemyModifierFn + 注册
  3. factories/spawnEnemy.ts 自动处理 tier 缩放 / elite buff / time scaling
  4. 完成

加一个 Tome（被动）

  1. types.ts TomeType + config.ts TOME_MAX_LEVELS 加新条目
  2. data/tomes.ts 新增 TomeDef：影响 stat 的返回 modifier 列表；上下文型（thorns / luck / xp_gain ...）modifiers: () => [] + contextOnly: true
  3. stats/recomputePlayerStats.ts 自动遍历 tome modifiers + finalize → 写回 player
  4. contextual tome 在对应代码路径读 player.tomes.find(...)?.level

加 i18n 文本

  1. i18n/en.json 加键
  2. i18n/zh.json 加键
  3. 代码用 t('your.key')

⚠️ 实际效果与文案差异

shield_tome 实际是双重减伤
不仅 +2 护甲/级（data/tomes.ts），受伤后还会再 ×(1 − 0.05·level)。
Lv 5 = +10 护甲 且 残余伤害 ×0.75。
curse_tome 是风险/收益型
堆满后：敌人速度 +30%、刷新间隔 −30%、击杀 XP +60%、拾取 XP 不变。
spawnInterval × max(0.5, 1 − 0.10·level)

永久商店（8 种）

跨局保留，用银币购买。来源：SHOP_UPGRADES (shop.ts:16)

战斗公式（乘法栈）

代码里武器伤害 / 受伤 / 攻速 / XP 都经过多层乘法叠加，下面把每条乘法栈摊开。

① 玩家造成伤害

stats.damage × player.damageMultiplier角色基础 + 商店 +5%/级 × isCrit ? critDamage : 1角色 + 商店 + precision → round(...) finalDamage

② 玩家受到伤害

enemy.damage − player.armor角色 + shield ×2/级 + 商店 → max(1, ...) × (1 − shield_tome.lv × 0.05)shield_tome 二次减伤 → round(...) finalDamage

③ 攻速

weapon.cooldownTimer −= dt × player.attackSpeedMultiplier1.0 + attack_speed_tome.lv × 0.1 → ≤ 0 触发开火

④ XP 收益（4 层乘法）

基础 XP × 1 + xp_gain_tome.lv × 0.15拾取阶段 × 1 + curse_tome.lv × 0.20击杀阶段 × 商店 xp_gain+10%/级 × 1 + min(combo × 0.05, 1.0)连击系统 → 最终 XP

⑤ 银币结算（局末）

baseSilver  = floor(killCount × 0.5 + level × 5)
victoryBonus = phase === 'victory' ? 100 : 0
runSilver   = silverEarned                        // 局内拾取/宝箱/任务
totalSilver = round((baseSilver + victoryBonus + runSilver) × tier.silverMultiplier)

祭坛 / Boss / 传送门 / Overtime

设计文档：docs/boss-loop-redesign.md · 实现：systems/altars.ts · systems/overtime.ts · systems/tierTransition.ts

核心循环：开局生成 1 个祭坛 → 玩家探索找到 → 按 [E] / 移动端按钮召唤 → Boss 战 → 击败后祭坛变传送门 → 玩家选择进入下一关（tier++）或留下打 Overtime。

祭坛状态机（5 阶段）

ready          玩家在交互半径内 → UI 显示 [E] 召唤 Boss
  ↓ 按 interact
summoning      读条 1.0s（防误触）；走出半径 → 回 ready
  ↓ 读条满
boss_active    Boss 已生成；祭坛锁住、不可再交互
  ↓ Boss 死亡
portal_ready   祭坛变传送门；UI 显示 [E] 进入下一关
  ↓ 按 interact
portal_used    终态；tier 推进流程消费它（清场 + 重生新祭坛）

Overtime 系统

玩家击败 Boss 后若不进入传送门：当 gameTime ≥ REGULAR_GAME_DURATION (540s)，state.overtimeSeconds 开始累加，敌人按时长升档。

常量值含义
REGULAR_GAME_DURATION540s常规生存期，超过后进入 overtime 计算
OVERTIME_STEP_SECONDS30s每多少秒升一档系数
OVERTIME_HP_DAMAGE_PER_STEP+10%每档敌人 HP / 伤害线性加成
OVERTIME_SPEED_PER_STEP+4%每档敌人速度增量（更温和，避免风筝失灵）

公式（factories/spawnEnemy.ts）：step = floor(overtimeSeconds / 30) → hp ×= 1 + 0.1 × step，speed ×= 1 + 0.04 × step。仅在 wave / miniBoss 模式下叠加；necromancerSummon / bossSummon 不受影响。

Tier 推进（进入传送门）

玩家在 portal_ready 阶段按 [E] → 祭坛切到 portal_used → tickTierTransition 在本帧末尾消费它：

  • config.tier++（最高 3，已是 3 时只重置时间）
  • 清场：enemies / projectiles / pickups / boss / damageEvents / waveIndex / finalSwarm / overtimeSeconds = 0
  • gameTime = 0（进入新一关的"常规生存期"）
  • 重新生成祭坛 + 宝箱
  • 保留：玩家 HP / 武器 / 典籍 / 等级 / XP / silver
  • phase = 'playing'

结算条件

结束时机触发结算
Boss 战死亡player.alive=false 且 phase='boss_fight'defeat，无 victory bonus
Overtime 死亡死时 overtimeSeconds > 0defeat（结算时可读 overtimeSeconds 区分死法）
Tier 3 通关后再进传送门真正的多关内容上线后再细化victory（保留 enum，目前未达到）

充能神殿（Charge Shrine）

灵感来自原 megabonk 的 Charge Shrine —— 站立读条解锁 → 4 选 1 永久增益。开局即可触发，与 Boss / 传送门流程独立。

核心循环：开局生成 3 座圣殿 → 玩家连续站满 4 秒 → 主循环暂停弹出 4 张奖励卡 → 玩家点选一张 → 永久写入 PlayerState → 圣殿变灰。

1. 配置常量 config.ts

常量值含义

SHRINE_RADIUS2.5玩家进入读条圈的半径（单位）
SHRINE_CHARGE_DURATION4.0充满 / 解锁需站立的连续秒数
SHRINE_REWARD_COUNT4解锁后的奖励选项数

2. 状态机 ShrinePhase

inactive ─(开局)─▶ charging ─(站满 4s)─▶ ready ─(玩家选完)─▶ consumed
                       ▲                  │
                       └─(玩家离开归零)──┘
  • charging：玩家在 2.5m 内累计 chargeTimer；离开立即归零（须连续）
  • ready：4 个奖励选项 roll 出来；GameState.phase = 'shrine_reward'，主循环全暂停（与 level_up 同语义）
  • consumed：玩家选完，永久消耗，圣殿变灰

同一时刻只允许一座圣殿处于 ready / shrine_reward，避免 phase 冲突；其他充满的圣殿排队等待。

4. 数值滚点 rollShrineOptions

基础稀有度权重: common 60 / uncommon 28 / rare 10 / legendary 2

luck 偏移（每 5% luck = +1 等级）:
  common    权重 -= 2 * luckLevel * 5  (下限 20)
  rare      权重 += luckLevel * 5
  legendary 权重 += luckLevel * 5

luck 来源 = luck_tome.level + floor(player.luckBonus * 100)

每选定 1 个稀有度，再在该稀有度池里加权随机抽取一条；同一个圣殿内不重复奖励类型。

4 层 stat 管线（PoE 风格）

公式: final = (base + Σadded) × (1 + Σincreased) × Π(more)

层语义例子
base基础值，setBase 覆盖式武器 lv 1 sword damage = 12
added平加值之和shield_tome lv 5 → armor +10
increased百分比累加（多 increased 同桶相加）speed_tome lv 5 → moveSpeed +40%
more独立倍数（每个 more 各自相乘）暴击 → ×critDamage（如 1.5）

tag superset-AND：modifier 带 tags 时，query 必须包含全部才生效（如 {tags:['fire','spell']} 仅对同时带 fire+spell 的攻击生效）。

必须遵守

  1. Three.js 命名空间导入：import * as THREE from 'three'
  2. game/core/ 禁止导入 Three.js
