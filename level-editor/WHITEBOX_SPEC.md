# 白盒规范（Whitebox Spec）— Blender 关卡灰盒速查

> 配合 `docs/level-system-redesign.md` 使用。本表是你在 Blender 里**搭白盒（灰盒）时的硬约定**。
> ⚠️ LevelLoader 正在并行开发；**严格按本规范命名 / 摆放**，导出的 glb 落地后就能被正确读出。

---

## 0. 最重要的三件事

1. **地图是 120 × 120**，坐标范围 **±60**（中心在原点）。所有碰撞体（`col_`/`wall_`/`climb_`）控制在 **±55** 内。
2. 代码**只认物体名前缀**。`.001`/`.002` 后缀会被忽略，但前缀错一个字符就识别不到。
3. 导出 GLB 时**必须勾 `+Y Up`**，否则整关躺倒。

---

## 1. 坐标系转换（Blender Z-up → 游戏 Y-up）

| Blender | → 游戏 | 记忆 |
|---|---|---|
| Position **X** | `x`（左右） | 不变 |
| Position **Y** | `z`（前后）= **−Y** | **取负！** |
| Position **Z** | `y`（高度） | 高度 |

> 在 Blender 顶视图（Numpad 7）：X 向右，Y 向上（屏幕里的"上"= 游戏里的"北/-z"）。Z 是离地高度。

## 2. 白盒只需要 3 种「逻辑体积」+ 出生点

白盒阶段**不用摆任何美术模型**，只摆下面这些（用 **Mesh Cube**，方便你在 Blender 里看见灰盒）。

> **核心原则：所见即所得（WYSIWYG）。** 代码读的是每个盒子的**世界包围盒**（不是 Scale 数值），所以「你画的盒子在哪、多大、多高，游戏里就是那样」。位置/缩放怎么调都行，只要最终盒子的外形对就行。

### 2.1 `col_` — 可站立地面 / 平台（最常用）

一块盒子，**它的顶面 = 玩家站立高度**。

| 盒子外形 | 游戏含义 |
|---|---|
| XZ 水平范围 | 平台占地（中心 + 宽深，自动算） |
| **顶面 Y** | **可站立高度**（你站上去时脚的 y） |
| 厚度 | 随意，纯视觉（顶面位置才重要） |

> 例：地面竞技场 → 画一块 30×30、顶面在 y=0 的薄板。
> 例：y=2 的 8×8 站台 → 画一块 8×8、**顶面抬到 y=2** 的盒子。

> 🆕 **`col_` 现在是"实体盒子"**（不只是一个高度面）。代码读它的**底面~顶面**整段体积，碰撞行为接近真实游戏的箱子：
> - **顶面**：可站立（高差 ≤0.5 自动迈上）。
> - **侧面**：挡住水平移动（玩家半径 ~0.45，会贴边停住，**不再穿墙**）——除非顶面在迈步范围内（那就直接踩上去）。
> - **底面在头顶之上**（高架/悬空平台）：可以从**下面穿过**，不会变"空气墙"。
>
> 所以"实心建筑/柱子/矮墙"**直接用 `col_` 就能挡人**；`wall_` 用于"看不见的隐形墙 / 栏杆"等纯阻挡。
> 注意：代码用**世界坐标轴对齐包围盒（AABB）**算盒子——**别旋转** `col_/wall_/climb_`（斜着放 AABB 会比可见盒大，产生看不见的占地）。要斜墙就用没旋转的细长盒子拼。

### 2.2 `wall_` — 实心遮挡 / 掩体（挡水平移动，玩家穿不过）🆕

**你画的整个盒子 = 墙体积**。盒子的底面~顶面就是墙的 `bottomY~topY`。

> 例：一堵贴地、4 宽、2 高的掩体 → 画一个 4×(薄)×2 的盒子，底面贴 y=0、顶面到 y=2。

### 2.3 `climb_` — 攀爬体（玩家可爬上 / 爬一半跳下；怪物可经此登高）🆕

同 `wall_`：**你画的整个盒子 = 可攀爬体积**，盒子底面~顶面就是可攀爬的高度区间。

> 🆕 **`climb_` 现在也是实体**：直接走撞不过去（走不穿），只能**按跳抓住后攀爬**。所以攀爬面贴在墙/高台侧壁即可，自带阻挡。

> 例：让玩家从 y=0 爬到 y=4 → 在高台侧面贴一个底面 y=0、顶面 y=4 的薄盒子。

**操作方式（已实现）：**
- 靠近 `climb_` 时**按跳**抓住墙（或下落中撞到攀爬体自动贴上）。
- 抓住后：**按移动方向**向上爬，**按 slide 键**向下爬，**不按**则悬停。
- **爬到顶面**自动翻身站上平台（会朝当前朝向落到平台上——把 `climb_` 顶面对齐目标平台顶面）。
- 攀爬中**再按跳** = 蹬墙跳下（脱离墙、进入下落）。
> 建模要点：`climb_` 顶面 `topY` 要 ≈ 目标平台的可站立高度，玩家才能顺势翻上去；薄盒子贴在平台侧壁即可。

### 2.4 `ramp_` — 斜坡（**走**上去的倾斜地面，区别于攀爬）🆕

把盒子建成一块**薄的、倾斜的板**（斜面）。代码扫描它的顶点，自动判断：
- **坡度轴**（沿 X 还是 Z 升高）= 最低点到最高点的水平主方向；
- **低端高度 = 顶面最低处**、**高端高度 = 顶面最高处**；
- 顶面在两端之间**线性插值** → 玩家走在上面会跟着斜面上下（不是攀爬）。

> 建模要点：
> - 做成**薄斜板**（厚度小），否则用包围盒最低/最高点估高度会有误差。
> - **高端高度要对齐它连接的平台顶面**（差 ≤0.5），否则到坡顶会被平台侧面（实体）挡住上不去。
> - 斜坡是可行走面、**不挡横向**；怪物也会自动沿斜坡上下。

### 2.5 出生点 — 用 **Empty → Plain Axes / Arrows**

| 物体名 | 含义 | 数量 |
|---|---|---|
| `spawn_player` | 玩家出生点 | **必须 1 个** |
| `spawn_boss` | Boss 出生点 | **必须 1 个** |
| `spawn_altar` | 祭坛/传送门 | 0–N（建议 1） |
| `spawn_chest` 🆕 | 宝箱位置 | 0–N |
| `spawn_enemy_N/S/E/W` | 敌人刷新区 | 可选（不设→默认地图边缘） |

> 出生点只读 Position（按 §1 转换），Scale/旋转无所谓。

## 3. 高度与可达性（搭白盒时必须遵守）

> ⚠️ **重要变更（已与代码对齐）：加载关卡后没有"自动坡道"，也没有默认地板。** `col_` 是纯粹的"所见即所得地面"——只有盒子顶面正上方能站，盒子之外即虚空（会掉下去）。想上更高的面，只能靠 **①阶梯式 `col_`（每级差 ≤0.5）** 或 **②`climb_` 攀爬体**。

| 相邻可站立面高度差 Δh | 玩家怎么上去 | 白盒建法 |
|---|---|---|
| **≤ 0.5** | 走路自动迈上（无感） | 直接相邻 `col_`，差 ≤0.5 |
| **0.5 ~ 1.0** | 普通跳（跳到顶面落上去） | 平台顶面 ≤1.0，跳得上 |
| **1.0 ~ 1.44** | 兔子跳（落地 0.3s 内再跳） | 平台顶面 ≤1.44 |
| **> 1.44** | `ramp_` 斜坡 / 阶梯 / `climb_` | 见下 |

**上到更高层的三种正规做法：**
1. **`ramp_` 斜坡**（走上去）——薄斜板，高端对齐目标平台顶面。最自然。
2. **阶梯**——堆若干 `col_`，每级差 ≤0.5，走路无缝迈上。
3. **`climb_` 攀爬**（爬上去）——贴在垂直墙面，按跳攀爬。

> 注意：单个倾斜的 `col_`（不是 `ramp_`）会被当成**一个平整高台**（取包围盒顶面），不会变成斜坡。要斜坡就用 `ramp_`。

**高度层语汇**：建议沿用 y = 0 / 2 / 4 / 6。层与层之间用 `ramp_` / 阶梯 / `climb_` 连接，否则上不去。

## 4. 推荐白盒积木尺寸（Cube 半尺寸 Scale 速查）

> 用默认 Blender Cube（2m 立方），Scale 即"半边长（单位）"。

| 用途 | Scale (X,Y,Z) | 实际尺寸 |
|---|---|---|
| 大地块 | (15,15,0.1) | 30×30 板 |
| 走廊段 | (3,8,0.1) | 6×16 板 |
| 中台站台 | (4,4,0.1) | 8×8 板 |
| 小落脚点 | (2,2,0.1) | 4×4 板 |
| 矮掩体墙 | (2,0.3,1) | 4 宽×2 高墙 |
| 高隔断墙 | (0.3,4,2) | 8 长×4 高墙 |
| 攀爬面 | (2.5,0.3,2) | 爬 4 高 |

## 5. 一个最小可玩白盒（起步模板）

```
spawn_player   Empty   Pos(0, 30, 0)        # 玩家从南侧出生 (游戏 z=-30)
spawn_boss     Empty   Pos(0, 36, 0)        # 北侧 (游戏 z=-36)
spawn_altar    Empty   Pos(20, 0, 0)        # 东侧祭坛

col_ground     Cube    Pos(0,0,0)    Scale(20,20,0.1)    # 40×40 中央地面
col_ledge_n    Cube    Pos(0,-25,2)  Scale(4,4,0.1)      # 北侧 y=2 站台(注:Blender Y=-25→游戏 z=25? 见下)
wall_cover_e   Cube    Pos(8,0,1)    Scale(2,0.3,1)      # 东侧掩体
climb_n        Cube    Pos(0,-21,1)  Scale(2.5,0.3,1)    # 爬上北站台
spawn_chest    Empty   Pos(-15,15,0)
```

> 注意 §1：Blender 的 `Pos Y` 会变成游戏 `-z`。先在 Blender 顶视图按"屏幕布局"摆，导出后游戏里前后会镜像，习惯就好；或导出后在游戏里实测位置再回 Blender 微调。

## 6. 导出设置

1. 选中关卡集合里**所有物体**（col_/wall_/climb_/ramp_/spawn_/将来的模型）。
2. `File → Export → glTF 2.0 (.glb)`。
3. 右侧：**Format = GLB**、**Include = Selected Objects**、**Transform → +Y Up ✅**、**Apply Modifiers ✅**、**Animation ❌**。
4. 输出到 `public/models/levels/level_whitebox.glb`。

> ⚡ **测试**：在 URL 加 `?level` 加载默认 whitebox（或 `?level=foo` 加载 `level_foo.glb`）。
> 控制台会打印 `[Level] Loaded (mode) ...` 其中 mode = `two-file` / `visual-only` / `col-only`。
> 文件不存在时自动回退到内置 Neon Crucible。
> 当前已接入：`col_`（实体地面/盒子，盒外为虚空会掉落，**倾斜的 col_ 自动当斜坡**）+ 迈步/掉落/支撑面跟随 + `wall_`（横向遮挡）+ `climb_`（实体+攀爬）+ `ramp_`（可行走斜坡）+ `spawn_player`（出生）+ `spawn_boss`（Boss 出场点）+ `spawn_altar`（祭坛）+ `spawn_chest`（宝箱）+ 全场景白盒渲染。
> 暂未接入：`spawn_enemy_*`（已解析+打印，敌人目前仍从地图边缘刷新）；怪物绕墙寻路尚未做（敌人直线追，但会沿斜坡/地形高度走）。

### 6.1 双文件模式（推荐生产用）🆕

为了让**视觉高模**（精致美术）和**碰撞低模**（精确碰撞）解耦，加载器支持「双文件」约定：

```
public/models/levels/
├── level_${name}.glb        视觉高模（玩家看到的关卡）
└── level_${name}_col.glb    碰撞低模（只含 col_/wall_/climb_/ramp_/spawn_*）
```

**加载策略**（`?level=${name}`）：

| 文件状态 | mode | 视觉来源 | 碰撞来源 |
|---|---|---|---|
| 两个都在 | `two-file` | `level_${name}.glb` | `level_${name}_col.glb` |
| 只有 visual | `visual-only` | `level_${name}.glb` | `level_${name}.glb`（同源解析） |
| 只有 col | `col-only` | `level_${name}_col.glb` | `level_${name}_col.glb`（灰盒） |
| 都不在 | — | 回退内置 Neon Crucible | 同左 |

**双文件模式的好处：**
- 视觉文件可以是任意精度的高模（不影响碰撞精度）。
- 碰撞文件只含**几何意图明确的低模**（手工搭的方块/斜板），避免「自动从高模里推断碰撞」带来的误差（陡坡、旋转面对不齐 mesh 等）。
- 两边相互独立 → 视觉迭代 / 碰撞调优互不干扰。

**双文件模式约定：**
- **碰撞文件里的所有几何**（`col_/wall_/climb_/ramp_`）会被解析；**视觉文件里的同名前缀会被忽略**（避免双源冲突）。
- 出生点 `spawn_*` 也只从碰撞文件读 —— 出生点本质是逻辑，归碰撞侧。
- 视觉文件可以**完全没有** `col_/wall_/climb_/ramp_/spawn_*`，纯 mesh art。
- 命名约定 `_col` 后缀**不能改**，加载器会按这个名字探测。

**白盒阶段**：单文件就够了（visual-only 模式），不用拆。
**生产阶段**：建议拆双文件 —— 美术做高模时不用顾忌碰撞盒，关卡设计师改碰撞时不用碰美术。

### 导出前自检

- [ ] 有且仅有 1 个 `spawn_player`、1 个 `spawn_boss`
- [ ] 每块要站的地面都有 `col_`，相邻高度差遵守 §3 可达性
- [ ] 要挡人的墙用 `wall_`（不是只摆视觉柱子）
- [ ] y>1.44 的高台配了 `climb_` 或阶梯（每级 ≤0.5），不要指望自动坡道
- [ ] `spawn_player` 必须在某块 `col_` 顶面正上方（否则一出生就掉进虚空被传送回原点）
- [ ] `col_/wall_/climb_` 盒子**不要旋转**（AABB 会算大，产生空气墙）
- [ ] 所有 `col_/wall_/climb_` 在 ±55 内
- [ ] 勾了 `+Y Up`，文件 < 5MB

## 7. 验证标尺（建议在 Blender 里常备）

放几个参考体，肉眼校准 metric：

- **跳跃标尺**：一个高 1.0 的 Cube（Scale Z=0.5, Pos Z=0.5）→ 任何"要求跳上去"的台阶不能比它高。
- **兔子跳标尺**：高 1.44。
- **迈步标尺**：高 0.5 → 比它矮的高差走路就能上。
- **地图边界**：120×120 方框（±60），实际摆放收在 ±55 内。
- `level-editor/megabonk_asset_library.blend` 里 `_Reference` 集合已有 120×120 网格可对齐。

## 8. 白盒阶段不用管的

- 美术模型（platform/灯/招牌等）—— 灰盒先验证布局和走位，美术后面再贴。
- InstancedMesh / draw call —— 那是 LevelLoader 的事。
- 材质 / 贴图 —— 白盒纯灰即可。
